## Maintenance: OpenStack services Genestack 2025.4/OpenStack Caracal (2024.1) base (mostly) to Genestack 2026.1/OpenStack Epoxy (2025.1)

### Notes

* `(opt)` refers to `/opt`, the standard install path for Genestack.
* `(etc)` refers to `/etc`, the standard override path for Genestack.
* Replace all placeholder values in angle brackets before using this runbook, or read them as applying to the specific service.
* This runbook applies generically to chart-based Genestack OpenStack services going to the Genestack 2026.1 release. Generally, upgrades are performed by re-running the documented Helm install workflow for that component. The docs site is generated from the repo `docs/` tree, and the OpenStack service order is defined in the docs navigation. Follow the OpenStack service order in the docs sidebar rather than inventing an ad hoc sequence. ([GitHub][1])
* For chart-based services, the install scripts in `(opt)/bin` read the target chart version from `(etc)/helm-chart-versions.yaml`, layer values from `(opt)/base-helm-configs/<component>`, `(etc)/helm-configs/global_overrides`, and `(etc)/helm-configs/<component>`, resolve required Kubernetes secrets, and pass any extra CLI arguments through to `helm upgrade --install`. That is why this runbook uses `--dry-run --debug` on the install script before the live apply. ([GitHub][6])
* Do **not** use this generic runbook for:

  * **Neutron** — use `maintenances/maintenance-neutron-2024-1-to-2025-1-epoxy.md`. ([GitHub][2])
  * **Skyline** — use `maintenances/maintenance-skyline-2024.1-to-2025.1-helm-migration.txt`. Skyline is a migration from kustomize to Helm and has service-specific secret and routing changes. ([GitHub][7])
* **Cinder** is mostly generic, but if the environment uses bare-metal `cinder-volume` workers, run the documented Cinder volume playbook after the chart install. The LVM iSCSI doc is explicit that `cinder-volume` runs on bare metal outside Kubernetes. ([GitHub][5])
* Do **not** rerun `(opt)/bin/create-secrets.sh` as a normal upgrade step. The script refuses to overwrite an existing `(etc)/kubesecrets.yaml` and warns that doing so would cause mass secret rotation. Use the dry-run to identify missing or changed secrets, then fix only those secrets. ([GitHub][8])
* The published `docs/release-notes.md` file is generated by Reno. For upgrade prep, search `(opt)/releasenotes/notes` directly and read every matching note for the component before touching the live chart. ([GitHub][3])

### Validation

* **Validated source version:** Genestack 2025.4, OpenStack Caracal (2024.1) base
* **Validated target version:** Genestack 2026.1, OpenStack Epoxy (2025.1) base
* **Upgrade path:** for most chart-based OpenStack services, direct re-run of the install script is the normal path. Neutron and Skyline are exceptions with dedicated service-specific maintenance. ([GitHub][1])
* **Major operational risks:** missing runtime secrets, release-note-driven config changes not applied, missing database backup before DB migration work, incomplete Cinder bare-metal post-step, and accepting a “helm succeeded” state without verifying all pods/jobs actually reconciled. ([GitHub][6])

### Goal

Upgrade `<component-name>` from `<from-version>` to `<to-version>` for Genestack 2026.1 without service regression, with the chart in a normal deployed state, and with all affected workloads healthy after the change.

### Prep

## Deployment Node

Use the standard deployment host or bastion that has:

* the target `(opt)/genestack` checkout,
* `(etc)/genestack` overrides,
* working `kubectl`, `helm`, and `yq`,
* `ansible` if the component is Cinder with bare-metal volume workers,
* and the usual OpenStack operator CLI access.

Verify the local checkout of /opt is on the approved Genestack 2026.1 ref before continuing.

If the checkout is not already on the approved 2026.1 ref, stop and align the checkout first.

Verify the install script exists:

```bash
test -x /opt/genestack/bin/install-<component-name>.sh
```

Expected: exit code `0`.

If the script does not exist, stop and reassess. This runbook only applies to chart-based services with a Genestack install script. ([GitHub][6])

Verify current component health:

```bash
helm list -n openstack --filter '^<component-name>$'
helm status <component-name> -n openstack
kubectl -n openstack get pods -o wide | grep -w '<component-name>'
```

Expected:

* the Helm release exists,
* `helm status` reports `STATUS: deployed`,
* current pods are healthy enough to establish a known-good baseline.

Verify current cluster or platform health:

```bash
kubectl get nodes
kubectl get pods -A | grep -E 'CrashLoopBackOff|Error|ImagePullBackOff|CreateContainerConfigError|Pending'
kubectl -n openstack get events --sort-by='.lastTimestamp' | tail -n 50
```

Verify the current deployed version:

```bash
helm list -n openstack --filter '^<component-name>$'
grep -E "^[[:space:]]*<component-name>:" /etc/genestack/helm-chart-versions.yaml
```

Expected:

* the currently deployed release is understood,
* the current `(etc)/helm-chart-versions.yaml` entry is visible,
* the operator can confirm that the environment is actually starting from `<from-version>`.

If the current version does not match `<from-version>`, stop and reassess.


Verify backups, snapshots, or restore points are available.

For any DB-backed or schema-migrating OpenStack service, treat a fresh MariaDB backup as mandatory before the upgrade. This generally only works from a node in the cluster, not an overseer type node. The repo currently contains the backup script at `(opt)/scripts/backup-mariadb.sh`; the docs also mention `(opt)/bin/backup-mariadb.sh`. In any case, take a database backup before proceeding.

Expected:

* a successful backup completes

If backups are required but missing, stop and create them before continuing.

## Configuration Review

Identify the configuration files and values that control this component:

```text
/etc/genestack/helm-chart-versions.yaml
/opt/genestack/bin/install-<component-name>.sh
/opt/genestack/base-helm-configs/<component-name>/*.yaml
/etc/genestack/helm-configs/global_overrides/*.yaml
/etc/genestack/helm-configs/<component-name>/*.yaml
/etc/genestack/kustomize/<component-name>/overlay/
/opt/genestack/releasenotes/notes/*
```

Verify the current config:

```bash
ls -1 /opt/genestack/base-helm-configs/<component-name> 2>/dev/null
ls -1 /etc/genestack/helm-configs/<component-name> 2>/dev/null
```

- Find the release notes.

Expected:

* the operator can see every values layer that will affect the chart,
* the operator has read all relevant release-note files for the component.

If any non-standard override exists, document it in the maintenance log before continuing.

## Pre-Change Safety Checks

Check for unhealthy pods, jobs, or dependent services:

```bash
kubectl -n openstack get pods
kubectl -n openstack get jobs
kubectl -n openstack get pods | grep -E 'mariadb|rabbitmq|memcached|keystone'
helm status <component-name> -n openstack
```

Check for open alerts or known blockers:

```bash
kubectl -n openstack get events --sort-by='.lastTimestamp' | tail -n 50
kubectl -n openstack get events --field-selector type=Warning
```

If any critical dependency is unhealthy, stop and resolve it first.

### Execute

## Validate the Target Version

Check the version source of truth: /etc/genestack/helm-chart-versions.yaml

If this maintenance requires an intermediate version, perform each hop separately and validate after each hop.

## Apply Required Overrides or Patches

There is **no single generic override block** for all OpenStack services in this maintenance. The engineer must read the component’s release notes and update only the service-specific overrides that the release notes and local environment require. That is a required decision point for this runbook. ([GitHub][10])

Update the required override file, if needed:

```text
/etc/genestack/helm-configs/<component-name>/<component-name>-helm-overrides.yaml
```

If no overrides are required after review, record that explicitly in the maintenance log.

## Run the Maintenance

Clear cached charts first, as recommended in the Genestack upgrade doc. ([GitHub][1])

```bash
find /etc/genestack/kustomize/ -name charts -type d -exec rm -rf {} \;
```

Run a dry-run or blast-radius check first:

```bash
/opt/genestack/bin/install-<component-name>.sh --dry-run --debug
```

Expected:

* the Helm render completes,
* no secret lookup fails,
* no values file or post-render overlay error occurs,
* no obvious chart validation error appears.

If the dry-run fails on missing secrets, identify the exact required secrets from the install script and create or fix only those secrets. Do not mass-rotate secrets by rerunning `create-secrets.sh`. ([GitHub][6])

Helpful secret check:

```bash
grep -oE "get secret [^ ]+" /opt/genestack/bin/install-<component-name>.sh | awk '{print $3}' | sort -u | \
  xargs -r -I{} kubectl -n openstack get secret {}
```

Run the maintenance command:

```bash
/opt/genestack/bin/install-<component-name>.sh
```

Previously completed jobs sometimes get in the way here and you may need to delete those for the installation to proceed.

### Cinder-only post-step

If `<component-name>` is `cinder` **and** the installation uses bare-metal `cinder-volume` workers, run the appropriate Cinder playbook after the chart install. The exact backend vars and limits are installation-specific and must be filled in for the site. The docs confirm that this playbook step is required for bare-metal `cinder-volume`. ([GitHub][11])

Example pattern:

```bash
ansible-playbook -i /etc/genestack/inventory/inventory.yaml \
  /opt/genestack/ansible/playbooks/deploy-cinder-volume.yaml \
  -e "@/etc/genestack/ansible-ext-vars/cinder-volume-reference-external-vars.yml" \
  -f 5 \
  --limit <cinder-storage-node-group>
```

Notes:

* Use **5–10 forks**, but never more than the number of storage nodes if the node count is lower.
* Adjust `--limit` and `-e` files to the actual installation being changed.
* Run any additional Cinder backend playbooks required by the service documentation for that installation.

### Post-Maint

Verify the deployed version:

```bash
helm list -n openstack --filter '^<component-name>$'
grep -E "^[[:space:]]*<component-name>:" /etc/genestack/helm-chart-versions.yaml
```

Expected:

* Helm shows the component release in a normal deployed state,
* the configured chart version matches `<to-version>`.

Verify workload health:

```bash
kubectl -n openstack get pods -o wide | grep -w '<component-name>'
kubectl -n openstack get jobs | grep -w '<component-name>'
helm status <component-name> -n openstack
```

Expected:

* pods are `Running`,
* jobs are `Completed`,
* `helm status` is still `deployed`.
* verify restart count is not incrementing

Verify dependent services:

```bash
kubectl -n openstack get pods | grep -E 'mariadb|rabbitmq|memcached|keystone'
```

Verify logs or events for upgrade failures:

```bash
kubectl -n openstack get events --sort-by='.lastTimestamp' | tail -n 50
kubectl -n openstack logs <primary-pod-or-deployment-for-<component-name>> --all-containers=true --tail=200
```

Verify user-facing functionality:

```bash
# Run a component-appropriate smoke test.
# Examples:
# keystone  -> openstack token issue
# glance    -> openstack image list
# nova      -> create and delete a small test server
# cinder    -> create and delete a small test volume
# horizon   -> UI login (If installed)
# skyline   -> UI login / API health
# heat      -> create and delete a trivial stack
```

Expected:

* the service works from an operator or user perspective,
* the engineer is satisfied that the service-specific workflow still functions.

Do not stop at “pods are Running.” Confirm that pods which should have restarted actually reconciled to the new revision and that no old pod instances are still serving.

Re-enable anything disabled during prep:

```bash
# Re-enable muted alerts, maintenance silences, paused routes, or any temporary controls used for the window.
```

### Troubleshooting

## Common Failure Signal

The most likely generic failure indicators are:

* dry-run or live apply fails with a missing Kubernetes secret,
* DB sync or Keystone user jobs fail,
* pods remain in `CrashLoopBackOff`,
* or the chart finishes but the service-specific smoke test fails.

These are the main reasons this runbook requires both release-note review and a mandatory dry-run first. ([GitHub][6])

## Rollback

Generally, you will want to roll forward, as components may perform database schema migrations. Rollback might involve a database restoration, and isn't recommended. You may attempt to proceed by chart rollbacks and database restores.

Important:

* If the upgrade executed irreversible DB migrations, do **not** assume `helm rollback` alone is enough.
* Be prepared to restore the DB from the pre-change MariaDB backup if rollback requires schema/data recovery. The MariaDB operations doc contains the restore workflows. ([GitHub][4])

## Additional Recovery Actions

If pods remain stuck:

```bash
kubectl -n openstack delete pod <stuck-pod-name>
kubectl -n openstack get pods -o wide | grep -w '<component-name>'
```

If settings do not reconcile:

```bash
helm get values <component-name> -n openstack
kubectl -n openstack rollout restart deployment/<deployment-name>
```

If the component is Cinder and bare-metal volume workers did not reconcile:

```bash
ansible-playbook -i /etc/genestack/inventory/inventory.yaml \
  /opt/genestack/ansible/playbooks/deploy-cinder-volume.yaml \
  -e "@/etc/genestack/ansible-ext-vars/cinder-volume-reference-external-vars.yml" \
  -f 5 \
  --limit <cinder-storage-node-group>
```

If the component is Neutron or Skyline, stop using this generic runbook and move to the dedicated service maintenance document. ([GitHub][2])
